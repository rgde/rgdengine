Правила оформление кода

Именование типов:

  • Классы начинаются с большой буквы C – CClass, CObject, CSomeObject
  • Интерфейсные классы (в которых все методы чисто виртуальные " = 0") – ISomeInterface, IBaseType
  • Имя структуры начинается с S – SSomeStructure, SDataType
  • Имя темплейтного класса начинается с T – TTemplateClass, TVector, TEvent
  • Имя enum типа начинается с E

Именование глобальных ф-ий и ф-ий мемберов:

  • Ф-ия мембер начинается с маленькой буквы
  • Ф-ия мембер начинается с прописной буквы если она статическая
  • Имя глобальной ф-ии начинается с прописной буквы
  • Имя любой ф-ии должно начинаться с глагола

Примеры правильно именования:
Class CSomeClass
{
int getValue();
static void HideAll():
}
 
void FreeUnusedMemory();

Имена переменных:

  • переменная член класс должна начинаться с m – m_nSomeMemberVal;
  • если переменная статическая у ее есть префикс s – ms_nSomeStaticMemeberVar;
  • после символа «_» должна следовать информация о типе так:
    • int, unsigned, _size-t – n (nSomeIntVal)
    • float, double – f – fSomeFloatValue
    • string, char* - str – strSomeStringValue
    • pointer – p -  MyObj* pMyObj;
    • std::vector<type> - ar or vec
    • std::list<> - list
    • …
    • класс – с - CSomeClass  cVariable
    • ссылка – r – CSomeClass &rVariable
    • std::map<> - map – std::map<int> mapVariables
    • handle – h – HWND hWnd

Каждый public класс и каждый public метод, назначение которых неочевидно из имени или имеющий специальные требования по работе с ним,  должен содержать комментарий Doxygen формата. Например

/// класс для демонстрации правил оформления кода

class MyClass
{
/// этот метод никогда не будет вызван
void MySuperPuperMethod();
};

или так

/** краткое описание  <точка>
 *   подробное описание (многострочное)
 *   при необходимости используйте директивы @todo <comment>, @author <name>
 */
class MyClass
{
void MySuperPuperMethod();
};

В начале каждого H файла может быть заголовок  Doxygen формата для пояснения назначения файла. Например

//////////////////////////////////////////////////////////////////////////
// @author <name>,  
// @date дата создания
// email:
// project: RGDE
// description: 
// example of use:
// @todo <comment>
//////////////////////////////////////////////////////////////////////////

Обработка ошибок

  • обработка ошибок должна быть
  • по возможности восстановление должно быть локальным (большинство случаев)
  • если клиент класса должен быть "предупрежден" о возможности ошибки - интерфейс должен содержать путь как проверить состояние ошибки до действия, которое к ней приводит
  • необходимо отделять основной путь исполнения от обработки ошибок а не смешивать их
  • на критическом по производительности пути ошибки, от которых возможно восстановление, должны сообщаться посредством кодов ошибок. Для кодов ошибок, которые не должны быть проигнорированы другими разработчиками можно использовать шаблон ErrorCode в base/exceptions.h
  • ошибки, от которых невозможно локальное восстановление и не допускают обработку на вызвавшем (верхнем) уровне должны сообщаться посредством исключений
  • на использование исключений наложены следующие ограничения:
    1. вместо оператора throw следует использовать фукцию throw_ из base/exceptions.h
    2. тип исключения должен быть классом унаследовавнимся от base::CBaseException из base/exceptions.h и называться с Exception на конце. 
    3. запрещается в явном виде использовать макросы __FILE__ и __LINE__ для передачи аргументов в конструктор этого класса. Взамен используйте макросы M_CBASEEXCEPTION_ARGS из base/exceptions.h
